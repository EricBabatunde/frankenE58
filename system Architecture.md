# Franken-E58: Full System Architecture & Implementation Blueprint

This reference document outlines the complete hardware topography, software data flow, and phased development lifecycle for upgrading the Mavic E58 frame into an ESP32-S3-powered, closed-loop stabilized micro-quadcopter.

---

## 1. System Architecture Overview

The drone's architecture decouples raw high-current actuation from sensitive low-signal telemetry. The ESP32-S3 acts as the central compute core, executing multi-threaded operations across its dual processor cores to handle real-time control loops and telemetry concurrently.

---

## 2. Hardware Architecture & Circuit Topography

### Power Distribution Network (PDN)

The system is powered by a **1S LiPo/LiHV battery (3.7V - 4.35V nominal)**. Due to heavy voltage drop caused by the coreless motors, the power rails must be explicitly split:

* **High-Current Rail (Raw Battery Voltage):** Feeds directly to the positive terminal of all four coreless brushed motors.
* **Low-Noise Logic Rail (3.3V Regulated):** Fed via the **AP2112K-3.3 Ultra-Low Dropout (ULDO)** regulator. Crucially, a $10\,\mu\text{F}$ tantalum capacitor paired with a $100\,\text{nF}$ ceramic capacitor must sit at the input and output of the regulator to filter transient ripples before they reach the ESP32-S3, MPU-6050, and VL53L0X.

### Actuation Subsystem (Brushed ESC Shield)

Each of the four coreless motors is driven by an independent low-side switching circuit integrated directly onto the shield board:

* **Switching Element:** **AO3400A N-Channel MOSFET** (SOT-23). Gate terminals are connected to the ESP32-S3 PWM pins.
* **Gate Protection:** A $10\,\text{k}\Omega$ resistor ties each gate terminal to Ground. This forces the MOSFETs to stay fully off when the ESP32-S3 boots up or flashes code.
* **Inductive Snubbing:** An **SS14 Schottky Diode** is placed in parallel across the motor terminals (Cathode to Battery Positive, Anode to MOSFET Drain) to suppress voltage spikes generated by the motor's coils.
* **EMI Suppression:** A $100\,\text{nF}$ ceramic capacitor is soldered directly across the motor pads to mitigate high-frequency brush-sparking noise.

### Sensor Bus Topology

The **MPU-6050 (IMU)** and **VL53L0X (ToF)** share a single $I^2C$ bus.

* **Bus Pull-ups:** Two physical $2.2\,\text{k}\Omega$ resistors pull the SDA and SCL lines up to the 3.3V logic rail.
* **Physical Layout Constraint:** The MPU-6050 must be physically isolated from the high-current power traces of the MOSFETs. The $I^2C$ trace lines must run over a solid ground plane to prevent inductive coupling from the motor paths.

### Microcontroller Pin Mapping (ESP32-S3 WROOM)

| Component | Target Pin | Pin Type | Critical Engineering Constraint |
| --- | --- | --- | --- |
| **Motor 1 (Front Left)** | GPIO 4 | Output (PWM) | Avoid boot strapping pins (GPIO 0, 45, 46). |
| **Motor 2 (Front Right)** | GPIO 5 | Output (PWM) | Avoid boot strapping pins (GPIO 0, 45, 46). |
| **Motor 3 (Rear Left)** | GPIO 6 | Output (PWM) | Avoid boot strapping pins (GPIO 0, 45, 46). |
| **Motor 4 (Rear Right)** | GPIO 7 | Output (PWM) | Avoid boot strapping pins (GPIO 0, 45, 46). |
| **I2C SDA** | GPIO 1 | Bidirectional | Connected to MPU-6050 and VL53L0X. |
| **I2C SCL** | GPIO 2 | Output (Clock) | Tied with external $2.2\,\text{k}\Omega$ hardware pull-ups. |
| **VL53L0X XSHUT** | GPIO 8 | Output (Digital) | Used to hard-reset and initialize the ToF sensor. |

## 3. Software Architecture & Data Flow

The software leverages the open-source **ESP-Drone** framework, running on top of **FreeRTOS**. It maps critical tasks across the ESP32-S3 dual-core processor to isolate deterministic control loops from asynchronous wireless operations.

```
       [ CORE 0: Network & Telemetry ]             [ CORE 1: Real-Time Flight Control ]
     +-----------------------------------+        +------------------------------------+
     |                                   |        |                                    |
     |  Wi-Fi AP <--> Web UI Control     |        |   Raw Gyro/Accel Data (MPU-6050)   |
     |  (LittleFS Host / nippleJS)       |        |                 |                  |
     |                 |                 |        |                 v                  |
     |                 v                 |        |     Hardware DLPF (94Hz/184Hz)     |
     |         Target Setpoints          |        |                 |                  |
     |    (Pitch, Roll, Yaw, Throttle)   |        |                 v                  |
     |                 |                 |        |       Biquad Notch Filters         |
     |                 +-----------------+--------+---->            |                  |
     |                                   |        |                 v                  |
     |                                   |        |    Extended Kalman Filter (EKF)    |
     |                                   |        |    (Fuses ToF Z-axis + IMU Data)   |
     |                                   |        |                 |                  |
     |                                   |        |                 v                  |
     |                                   |        |         PID Control Loops          |
     |                                   |        |                 |                  |
     |                                   |        |                 v                  |
     |                                   |        |     Mixer -> MCPWM (20-32kHz)      |
     |                                   |        |                                    |
     +-----------------------------------+        +------------------------------------+

```

### Core Execution Assignments

* **Core 0 (Networking & I/O):** Runs the Wi-Fi Access Point stack, handles incoming UDP target setpoints from the web-based `nippleJS` joystick control client, and services HTTP requests from the internal LittleFS partition.
* **Core 1 (Flight Dynamics):** Runs the deterministic flight control pipeline at a dedicated update rate of **400Hz**.

### The Control and Filtering Pipeline

1. **Hardware Digital Low-Pass Filter (DLPF):** The MPU-6050 is configured via $I^2C$ to pass its raw data through its internal hardware low-pass filter set to **94Hz**. This cuts out high-frequency vibrations from the motor shafts before the data reaches the microcontroller registers.
2. **Software Biquad Notch Filtering:** The raw gyro arrays are processed through a series of cascaded software notch filters within the ESP-Drone main task loop. These target the exact resonant vibration frequency of the plastic E58 motor arms.
3. **Extended Kalman Filter State Estimation:** The filtered angular rates are piped into the EKF state matrix. Concurrently, the Z-axis distance reading from the VL53L0X ToF sensor is introduced at 50Hz. The EKF updates the drone's estimated linear velocity and height vectors, using the absolute metric distance from the laser sensor to prevent the double-integrated accelerometer calculations from drifting over time.
4. **PID Attenuation and Mixer:** The inner loop evaluates target orientation vs. estimated orientation, updating the PID controllers. The output values are distributed by the motor mixer matrix into individual throttle profiles, which are then converted into high-frequency **20kHz to 32kHz MCPWM signals** to drive the gates of the AO3400A MOSFETs.

## 4. Phased Implementation Plan

### Phase 1: Breadboard Prototyping & Telemetry Validation

* **Objective:** Validate sensor communication and network performance before designing hardware layout.
* **Steps:**
1. Wire the ESP32-S3 Development Board to an external MPU-6050 and VL53L0X on a breadboard using $4.7\,\text{k}\Omega$ external pull-up resistors on the $I^2C$ bus.
2. Flash the ESP-Drone framework; open an $I^2C$ scanner script to ensure both sensor addresses register cleanly without bus timeouts.
3. Host the LittleFS file server container with the `nippleJS` joystick interface on Core 0. Measure the packet transmission delay via UDP to verify latency stays below **15–20ms**.



### Phase 2: Schematic Capture & PCB Design

* **Objective:** Design a compact, lightweight 2-layer or 4-layer custom shield board matching the internal mounting footprint of the E58 chassis.
* **Steps:**
1. Draw the complete electrical schematic including the AP2112K ULDO, four AO3400A low-side switches, flyback protection, and physical $I^2C$ pull-up circuits.
2. Execute component placement: Seat the ESP32-S3 module and the MPU-6050 at the exact geometric center of the board. Move the high-current MOSFETs and battery terminal connections to the board's periphery.
3. Rout the power distribution using wide traces (**minimum 40–50 mil width**) for the motor supply lines, keeping the signal trace widths for the $I^2C$ lines narrow (**8–10 mil**). Use a dedicated star-ground configuration to separate high-current return paths from logic ground.



### Phase 3: Bench Testing & Noise Profile Calibration

* **Objective:** Solder the custom shield board, assemble the electrical components, and map structural noise profiles.
* **Steps:**
1. Assemble the PCB shield and mount it inside the E58 body cavity. Secure the coreless motors into their respective housing sockets.
2. Strap the drone down safely onto a heavy structural bench mount (or a heavy weighted block).
3. Spin up the motors sequentially across a 0% to 100% throttle sweep while streaming raw, unfiltered gyro data over a serial connection.
4. Graph the resulting data via a Fast Fourier Transform (FFT) calculation to identify peak physical vibration frequencies. Use these specific frequency values to configure the software Biquad Notch Filter center-frequencies in your ESP-Drone configuration files (`config.h`).



### Phase 4: Mechanical Mount Engineering & Tethered Tuning

* **Objective:** Verify stabilization capabilities using 3D-printed components under controlled mechanical constraints.
* **Steps:**
1. Design and 3D print a downward-facing mount socket under the E58 belly pan to house the VL53L0X optical sensor, ensuring it has a clear, unobstructed vertical path to the floor.
2. Construct a single-axis gimbal tether rig (a string setup or mechanical pivot rod) that allows the quadcopter to rotate freely *only* along its Pitch or Roll axis while preventing free flight escape.
3. Activate the web controller and initiate a hover state. Manually induce physical disturbances to the frame and tune the **Proportional ($K_p$), Integral ($K_i$), and Derivative ($K_d$)** gains within the ESP-Drone system registers until oscillations decrease smoothly and the frame snaps quickly back to level.



### Phase 5: Untethered Flight Optimization

* **Objective:** Validate multi-sensor fusion performance during untethered, autonomous altitude-stabilized free flight.
* **Steps:**
1. Place the fully assembled drone on an open floor space away from obstacles.
2. Execute an automated takeoff sequence using the web interface joystick. The drone will lift off to an initial altitude of 0.5 meters utilizing the VL53L0X laser ranging data.
3. Verify altitude hold capability by placing a flat object under the drone's sensor path; the drone should react smoothly by rising to maintain its target height above the new surface.
4. Conduct fine-tuning of the EKF noise covariance parameters if any slow vertical drift or high-frequency motor twitching is observed during sustained hovers.
